Executive Summary
1.1 Platform Vision & Goals

AirTrainr is an intelligence-driven sports training marketplace designed to bridge the gap between athletes seeking professional coaching and certified trainers looking to grow their business. Operating across the United States and Canada, the platform supports 20+ sports disciplines and provides end-to-end management for the entire training lifecycle — from discovery and booking to payment and review.

The platform addresses three core problems: (1) athletes struggle to find qualified, local trainers for their specific sport and skill level; (2) independent trainers lack tools to manage their business professionally; and (3) there is no trusted payment and escrow system for training sessions in the sports coaching market.

Mission Statement: Democratize access to elite sports training by connecting athletes of all levels with verified, professional trainers in their area — while providing trainers with the tools to build and scale their coaching business.

Section Four
Business Workflows
End-to-end user journeys and system interactions
Digital Marketing Heroes
Workflows
4.1 Direct Booking Flow

1 Trainer sets availability (Recurring Weekly OR Per-Slot) 2 Athlete searches by sport/location 3 Views trainer profile with green calendar dates 4 Selects date, duration, sport, location 5 Stripe Checkout (price + 3% fee) 6 Webhook creates booking + payment_transaction (escrow) 7 Both receive email receipt 8 Session completed 9 Escrow released after hold period

4.2 Training Offer Flow

1 Trainer browses athletes 2 Creates offer (sport, date/time, rate, message) 3 Athlete receives notification 4 Opens offer modal (sees sessions, trainer info, rate) 5 Clicks “Accept” → redirected to Stripe Checkout 6 Payment confirmed → webhook creates booking + updates offer status 7 Both receive email receipts + notification

4.3 Camp Enrollment Flow

1 Trainer creates camp (name, per-day schedule, location, price, max spots) 2 Sends camp offer — dates/times auto-populated from camp schedule 3 Athlete sees read-only schedule preview 4 Accepts → Stripe payment 5 Webhook: fresh DB read → check spotsRemaining → if > 0, decrement + create booking → if = 0, auto-decline offer 6 Retry loop (3x) handles concurrent acceptances

Race Condition Protection: When two athletes accept the last camp spot simultaneously, the retry loop ensures only one succeeds. Each attempt freshly reads spotsRemaining from the database, checks if > 0, then decrements. If the write fails (stale data), it retries with a fresh read. After 3 failures, the operation is rejected.
4.4 Location Intelligence Flow

1 User types city name 2 Frontend calls /api/places/autocomplete (server proxy) 3 Google Places returns predictions (250ms debounce) 4 User selects city 5 /api/places/details returns full address (city, state, country, ZIP, lat/lng) 6 If country not in allowed_countries → “Not available in your area” banner + “Notify me” lead capture 7 ZIP auto-filled + cross-validated against city country 8 Travel radius auto-switches to km for Canadian postal codes

5.3 Stripe Connect Onboarding

Trainers connect their bank account through Stripe Express — a hosted onboarding flow managed entirely by Stripe. AirTrainr never sees, stores, or processes bank account details. The Payment Settings page shows three states:

Not Connected
“Connect Bank Account” button → creates Express account via /api/stripe/connect → redirects to Stripe hosted onboarding
Partially Connected
Amber warning → “Complete Setup” button → resumes Stripe onboarding where left off
Fully Connected
Green “Payouts Enabled” status → bank last4 + name displayed → Stripe dashboard link
Email Receipts
After every payment, both athlete and trainer receive HTML email receipts via GoDaddy SMTP with full cost breakdown, session details, and trainer payout amount.

Location
6.1 Google Places Integration

All location services are proxied through server-side API routes (/api/places/*) to keep the Google API key secure and avoid CORS issues. The LocationAutocomplete component provides a unified interface across all pages — registration, profile editing, trainer setup, and camp location.

FeatureImplementation
City AutocompleteServer proxy → Google Places Autocomplete → 5 predictions, 250ms debounce
Place DetailsOn selection → fetches city, state, country, ZIP, lat, lng from address_components
Country FilteringFetches allowed_countries from platform_settings → blocks results from non-allowed countries
Lead CaptureBlocked users see “Not available” banner → “Notify me” button → saves to location_leads table
ZIP ValidationCross-validates postal code format (US/CA regex) against city country from Google API
Radius UnitsAuto-detects Canadian postal codes → switches display to km. Stores internally as miles.
6.2 Country Access Management

Administrators can enable/disable countries from the Settings page. 15 countries are preconfigured with flags: US, CA, GB, AU, IN, DE, FR, BR, MX, JP, NZ, IE, ZA, AE, SG. The Location Demand dashboard aggregates leads from blocked regions, sorted by request count, helping inform expansion decisions.

Security
7.1 Security Architecture
LayerMeasureDetails
DatabaseRow Level Security (RLS)All 15+ tables have Supabase RLS policies. Users can only read/write their own data.
AuthenticationPassword + OAuth12+ char with uppercase, lowercase, number, special char. Google OAuth as alternative. No Apple login.
API SecurityRole verificationAll payment/connect APIs verify user exists AND has correct role (trainer) before processing.
Payment SecurityStripe ExpressBank details entered on Stripe hosted page. AirTrainr never sees/stores bank information. No Stripe account IDs exposed to frontend.
API Key ProtectionServer-side proxyGoogle Places API key never sent to browser. All calls proxied through /api/places/* routes.
Service RoleServer-onlySupabase service_role key used only in API routes + webhooks. Anon key for client with restricted RLS access.
Input ValidationMulti-layerClient-side (form validation) + server-side (API checks) + database constraints (CHECK, NOT NULL, foreign keys)
Webhook SecuritySignature verificationStripe webhook events verified with STRIPE_WEBHOOK_SECRET before processing.
